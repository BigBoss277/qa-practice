# Bug Report, Severity, Priority, and Defect Lifecycle

## Simple Explanation

A bug report is a structured document that describes a problem found in software. It usually includes a title, environment, steps to reproduce, expected result, actual result, severity, priority, and additional notes or attachments.

Severity describes how seriously a bug affects the product from a technical or user-impact perspective.

Priority describes how soon the bug should be fixed from a business or project perspective.

The defect lifecycle is the path a defect follows from discovery to closure. A common flow is: New -> Assigned -> Open -> Fixed -> Retest -> Verified -> Closed. Some defects can also be Reopened, Rejected, Deferred, or marked as Duplicate.

## Example From My Project

My projects include bug reports and issue examples. For example, in the login page project I documented that pressing Enter does not submit the login form.

Severity: Medium, because the user can still submit the form by clicking the button, but a common login behavior does not work.

Priority: High, because pressing Enter is a common user expectation and should be fixed soon.

I do not have a full real defect lifecycle in my projects yet because I am studying independently and do not have a developer team fixing the issues. However, I can still document the issue, classify it, and write retest notes as practice.

## Interview Answer

A bug report is a clear description of an issue that helps developers understand, reproduce, and fix the problem.

Severity is about the impact of the issue. Priority is about how soon the issue should be fixed.

For example, if a login form does not submit when the user presses Enter, I would classify it as Medium severity because login is still possible with the button, but High priority because it affects a common user workflow.

The defect lifecycle is the process a defect goes through from being found to being closed. In a real team, a defect can be reported, assigned, fixed, retested, verified, and closed.
